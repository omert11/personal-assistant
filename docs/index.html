Rules · 13 global kural

Her oturumda yüklenen davranış disiplini

ask-first
Önce Sor

Kullanıcıya yöneltilen her soru, seçim veya onay düz metinle değil, yalnızca AskUserQuestion tool çağrısıyla sorulur. Düz metin soru tamamen yasaktır.

Kural mutlaktır: tek bir evet/hayır onayı bile ("devam edeyim mi?" gibi tek satır metin yasak — yerine ["Evet","Hayır"] tool çağrısı); commit/destructive/geri-alınamaz onaylarda metinle "onaylıyor musun?" yasak; çoklu seçenekte markdown 1. 2. 3. liste yasak. Felsefe: varsayım = hata, sormak = kalite; çelişki görünce sor, bilgi eksikse tahmin etme.

Self-Check

Metne ? yazmadan önce dur: (1) kullanıcıya soru mu? → AskUserQuestion; (2) "A mı B mi?" fragment yasak; (3) "onaylıyor musun" formları hep tool. Önerilen seçenek ilk sırada (Recommended) etiketiyle.

coding
Kod Disiplini

İletişim Türkçe, kod yorumları ve commit mesajları İngilizce. Küçük ihlaller sessizce düzeltilir, mimari ihlaller durdurularak raporlanır.

Proaktif hijyen: unused import, typo, formatting sessizce düzeltilir; mimari ihlal (güvenlik açığı, yanlış pattern) görülürse işlem DURDURULUR ve AskUserQuestion ile raporlanır. Otonom çözüm: hata verildiğinde yönlendirme beklenmez. Geçici çözüm (workaround) yerine doğru çözüm bulunur. Hatalar her zaman wrap edilir.

Örnek
Go:     fmt.Errorf(": %w", err)
Python: raise X from err
TODO:   # TODO: hard-coded değer, placeholder, eksik validasyon
python
Paket Yönetimi

source .venv/bin/activate

workflow
Subagent + Background Çalıştırma

Sub-agent çağrısı kullanıcı onayı olmadan yasak; bloklayan veya süreli olabilecek HER iş (subagent, uzun Bash, workflow) ise her zaman background modunda çalıştırılır.

Kullanıcı "takım kur" derse TeamCreate. Subagent için açık izin yoksa önce AskUserQuestion sorulur ("hızlı olur/paralel" gerekçeleri geçersiz). İstisna: skill/agent-tanımı/hook akışının parçası olan çağrılar direkt yürür.

Çalıştırma kuralı bloklayan tüm işleri kapsar: subagent her zaman run_in_background: true (istisnasız); uzun/süreli Bash komutları (build, test, install, dev server, deploy/migration, log tail) background; workflow zaten background döner. Foreground'a düşmek yerine task-notification beklenir. Tek istisna: sub-saniye, yan etkisiz, çıktısı anında lazım olan kısa okuma komutları (ls, git status, version check) Bash'te foreground kalabilir.

obsidian
Bellek Kullanımı

Proje-spesifik kalıcı bilgi resmi Obsidian CLI ile okunur/yazılır; aramada obsidian-searcher, yapısal yazımda obsidian-writer agent'ı kullanılır.

Vault ~/Documents/ObsidianVault/<isim>/ altında MOC + wikilink yapısındadır. Arama elle yapılmaz — "önce MOC → files → BM25 search → context → oku → sentezle" akışını agent yürütür. BM25 literal arama (vector yok); tek/az terimle başla, TR↔EN sinonim dene, path= ile filtrele. Klasörler: index.md, Learnings/, Journal/.

Karar

Obsidian → proje-spesifik kalıcı bilgi · ~/.claude/memory/ → kullanıcı profili · ~/.claude/rules/ → tüm projeler.

init
Oturum Başı Kontrol

Her konuşmanın ilk mesajında proje yapılandırması (CLAUDE.md/local + Vikunja/Solo/Obsidian ID'leri) denetlenir; sorun bildirilirse Vikunja görevi kontrol edilir ve geçmiş Learnings ön-aranır.

5 madde kontrol edilir; eksikse "⚠️ Eksik Yapılandırma: /project-init" notu eklenir. Kullanıcı somut görev verince ve Obsidian Folder tanımlıysa, göreve başlamadan önce obsidian-searcher background çağrılarak geçmiş bilgi aranır (selamlaşma/kavramsal soruda tetiklenmez).

learnings
Bilgi Kalıcılaştırma

Kullanıcı tekrar eden bir komut/iş akışı/yapılandırma/bağlantı paylaştığında, bunu kural olarak kaydetmeyi AskUserQuestion ile teklif eder.

Değerli tekrar kullanılabilir bilgilerin uçup gitmesini önleyip /rule-manager ile kalıcılaştırmak için — header "Kural", options ["Evet, kaydet", "Hayır"].

before-commit
Teslimat Kapısı

Her commit/push/PR/branch teslimat isteğinde mutlaka commit skill'i çalıştırılır; manuel git commit yasaktır.

Skill teslimat öncesi tüm kontrolleri (code-review, test, rules uyumu, Vikunja, son onay) zaten yürütür — böylece tutarlı kalite kapısı her teslimatta uygulanır.

cli-tools
CLI Referansı

Sık kullanılan CLI araçlarının (Zed, Tailscale, UV, rsvg-convert, markitdown) komut referansını standartlaştırır.

Bir cheat-sheet kuralı: Zed dosya/satır/diff/--wait; Tailscale status/up/ping; UV venv/pin/install; rsvg-convert SVG→PNG/PDF; markitdown PDF/Office/ses/görsel/YouTube → markdown.

Örnek
rsvg-convert -w 512 -h 512 in.svg -o out.png
markitdown dosya.pdf -o cikti.md
zed --diff a.ts b.ts
browser-testing
Frontend Test

Tarayıcı otomasyonu ve frontend testi için MCP yerine Playwright CLI (playwright-cli) kullanılır.

CLI token-efficient ve Microsoft'un coding agent'lar için önerdiği yol — MCP playwright kaldırıldı. Snapshot + ref tabanlı etkileşim (a11y tree token maliyeti yok). Test suite için ayrıca @playwright/test.

Örnek
playwright-cli open https://example.com
playwright-cli snapshot
playwright-cli click e15
playwright-cli fill e20 "kullanici" --submit
b2c-booking-log
Domain Bilgisi

Diji B2C rezervasyon loglarının analizi için log dosyaları, DB modelleri, status akışı, logger hiyerarşisi ve bilinen sorunları haritalar.

Loglar server/logs/ (debug.log, diji.csv, modül csv'leri). Modeller: Booking, BookingItem, UserBookingBasket, AppLog. Status akışı: READY → PREBOOKED → PAYPROCESSING → PROCESSING → SUCCESS (hata: FAIL/CANCEL). Bilinen sorunlar: duplicate finalize race condition, PAYPROCESSING'de takılma, SMS NoneType hatası.

soloterm
Process Yönetimi

Tekrar eden shell komutları için mutlaka solo.yml oluşturulur ve komutlar solo CLI üzerinden process olarak yönetilir.

Bir komut ileride tekrar kullanılabilecekse önce solo.yml kontrol edilir; varsa Solo üzerinden çalıştırılır, yoksa önce eklenir. Process'ler solo processes / solo commands komutlarıyla başlatılabilir, durdurulabilir ve yeniden başlatılabilir.

⚠️ Process çıktısı/logu CLI'de YOK: Solo CLI ve HTTP API process loglarını göstermez (sadece status, pid, uptime). Crash sebebini görmek için Solo app GUI terminal paneli, 2>&1 | tee /tmp/<proc>.log ile loglama, curl ile traceback veya foreground Bash gerekir. (Bu kısıt CLI'da düzeltildiğinde uyarı geçersizleşir.)

token-efficiency
Model Ekonomisi

Subagent ve Workflow ajanlarında model her zaman bilinçli seçilir — en küçük yeterli model.

Agent frontmatter'ına her zaman açık model: yazılır: haiku (mekanik çıkarım/rapor), sonnet (yapısal yazım/çeviri/orchestration), opus/inherit (derin muhakeme). Workflow agent() çağrıları frontmatter kullanmaz — opts.model açıkça belirtilir; named plugin agent (subagent_type) çağrısında frontmatter geçerlidir. Maliyet: haiku ≈ Opus'un ~1/25'i, sonnet ≈ ~1/5'i — paralel fan-out'ta doğrusal çarpan.

Örnek
agent(prompt, { label: 'translate:ar', model: 'sonnet', schema })
Hooks · 8 hook, 3 event

Oturum yaşam döngüsü otomasyonu

SessionStart (6) · UserPromptSubmit (1) · Stop (1). Hepsi shell, bağımlılık minimal. Her hook ne yapıyor, nasıl çalışıyor ve ne kazandırıyor.

load-rules.sh
SessionStart

Oturum başında plugin'in rules/ klasöründeki tüm kural dosyalarını ~/.claude/rules/ altına materialise eder ve context'e ~19KB ham metin yerine yalnızca kısa bir indeks (dosya adı + tek satır özet) basar.

Nasıl çalışır

set -e fail-fast. Her .md için copy_if_newer: hedef yoksa veya kaynak daha yeniyse (-nt mtime) kopyalanır. Sonra "# Active Rules" başlığı + her kural için bir satır (ilk H2 → H1 → ilk satır, 80 char). Claude Code'un ~2KB hook preview truncation sınırını aşar.

Fayda

Kurallar her oturumda güncel ve sabit yoldan erişilebilir; context bütçesi şişmez — Claude ilgili kuralı gerektiğinde Read ile açar.

init-check.sh
SessionStart · + .ps1

CLAUDE.md/CLAUDE.local.md varlığını ve Vikunja/Solo/Obsidian Folder tanımlarını denetleyip tek satırlık sonuç (INIT_OK / INIT_MISSING / INIT_MALFORMED) basar.

Nasıl çalışır

hook-state.sh source edilip ortak hook_obsidian_folder yüklenir. Esnek pattern (Obsidian.*Folder) bitişik VE markdown-tablo formatını yakalar. Satır var ama değer çıkmıyorsa MALFORMED (sessiz tuzak yakalanır). Öncelik: eksik > bozuk > OK. Windows'ta init-check.ps1 karşılığı.

Fayda

Eksik/bozuk yapılandırma oturum başında yakalanır; entegrasyonlar sessizce devre dışı kalmaz, /project-init ile tamamlanır.

obsidian-context.sh
SessionStart

Projenin Obsidian bellek klasöründeki index.md MOC'unun frontmatter'ını ve başlıklarını oturum başında context'e dump eder.

Nasıl çalışır

hook_obsidian_folder ile klasör çıkarılır. <vault>/<folder>/index.md varsa awk ile frontmatter bloğu + grep ile tüm h1/h2/h3 başlıklar basılır, "detay için Read" notu eklenir.

Fayda

Claude oturuma proje belleğinin haritasıyla başlar; tüm vault'u okumadan doğru [[wikilink]]'i açabilir.

session-obsidian-daily.sh
SessionStart

Bugünün ve dünün global daily note'larından kısa özet basar; oturumun gerçek başlangıç zamanını state tablosuna damgalar.

Nasıl çalışır

session_id ile session_start damgalanır (Obsidian kapalı olsa bile — Stop hook'ların 5dk-yaş kontrolü buna bakar). IPC 2s timeout ile sarılır. daily:read kasıtlı çağrılmaz (boş dosya yaratıyordu) — dosya filesystem'den Daily/, Daily Notes/, root sırasıyla okunur, head -100.

Fayda

Oturumlar arası süreklilik — kullanıcı kaldığı yerden devam eder; session_start damgası Stop hook'ların doğru çalışması için kritiktir.

userprompt-obsidian-hint.sh
UserPromptSubmit

Her non-trivial prompt'a iki davranış enjekte eder: (1) ARAMA — KESIN KURAL: iş akışının herhangi bir anında ek efor gerektiren bir iş doğduğunda (sorun çözme, araştırma, doğrulama, log analizi, karar/komut/credential arama) önce obsidian-searcher ile vault'a bak; sonuç boşsa kendin yap ve bulguyu sonradan kaydet; (2) KAYIT — dar kriter: sadece bu oturumda YENİ öğrenilen credential/sunucu, çözülen non-trivial bug veya kalıcı mimari karar background kaydedilir; zaten yazılı bilgi, oturum özeti ve genel bilgi kaydedilmez, şüphede kaydetme; devam eden iş varsa kayıt tüm işler bitene kadar bekletilir ve kalan-iş raporuna dahil edilir.

Nasıl çalışır

Klasör çıkarılır, prompt <15 char ise skip. ARAMA her prompt'ta. KAYIT geciktirmeli: last_learning_hook'tan 5 dk geçmişse eklenir. HINT tek satır (jq --arg çok satırı escape etmez), hookSpecificOutput.additionalContext ile basılır. Eski stop-obsidian-remind.sh'in (block + "error" görünümü) yerini alır.

Fayda

Sıfırdan uğraşmadan önce geçmiş bilgi hatırlanır; kayda değer yeni bilgi rahatsız etmeden çıktıyla beraber kalıcılaşır — hem hatırlama hem öğrenme prompt akışına gömülür.

stop-obsidian-daily.sh
Stop

Oturum sonunda kayda değer aktivite varsa daily note'a proje-linkli tek satır özet append'ler. Sessiz çalışır, oturumda yalnızca 1 kez yazar.

Nasıl çalışır

Seyreltme: last_daily_hook doluysa atla. Oturum 5 dk'dan genç ise (session_start) atla. Son mesaj <200 char ise skip. Özet önceliği: customTitle (/rename) → aiTitle → ilk 280 char. Satır: - HH:MM [[folder/index|folder]]: özet — [transcript](file://...).

Fayda

Her anlamlı oturum, aranabilir ve transcript-linkli günlük kaydı bırakır; kısa/boş oturumlar gürültü yapmaz.

hook-state.sh
Altyapı · shared

Hook'ların oturum başına seyreltilmesi için TSV tabanlı state tablosu yönetir; tüm hook'ların paylaştığı hook_obsidian_folder çıkarım fonksiyonunu sağlar. (Hook değil, diğer scriptlerin source ettiği kütüphane.)

Nasıl çalışır

State: session_id ⇥ session_start ⇥ last_daily_hook ⇥ last_learning_hook. Saf awk (jq/sqlite yok), mkdir tabanlı atomik kilit lost-update'i önler, 7 günden eski satırlar budanır. hook_obsidian_folder bitişik/düz/tablo formatını destekler, placeholder/quote satırlarını eler.

Fayda

Tüm hook'lar tek kaynaktan (DRY) çalışır; harici bağımlılık olmadan macOS+Linux'ta güvenilir, kilitli, kendini temizleyen state mekanizması.

Skills · 14 adet

Tekrar eden işler için playbook'lar

Skill'ler ince tetikleyici/koordinasyon katmanıdır — ağır iş agent'lara delege edilir.

commit

Kod değişikliklerini commit etmeden önce kapsamlı kalite kapısı kurar. Tüm analizi (code-review, test, rules uyumu, Vikunja, Obsidian) sessizce yapar, bulguları biriktirir, tek toplu soru bloğunda sunar, onay sonrası seçilen teslimatı (commit/push/PR/merge) uygular.

Tetikleyici

"commit", "push et", "PR oluştur", "branch aç", /commit. before-commit kuralı gereği manuel git commit yerine zorunludur.

Akış

status → branch tespit → code-review (zorunlu) → test → rules tara → Vikunja eşleştir → Obsidian → tek soru bloğu → dosyaları tek tek stage + Co-Authored-By commit → teslimat.

Öne çıkan

Code-review atlanamaz — "hızlıca commit" dense bile çalışır. Bulgular küçük diye filtrelenemez/yumuşatılamaz; uncertain bulgular bile listelenir, false-positive kararını kullanıcı verir. Asla --amend, asla force push, asla git add -A.

extension-builder

Claude Code'u genişleten her bileşeni (skill/plugin/agent/hook/MCP/output-style/channel/marketplace) yazar. En kritik prensibi: schema/alan isimlerini bellekten asla uydurmaz — resmi dökümanı canlı çeker.

Tetikleyici

"skill yaz", "plugin oluştur", "hook ekle", "MCP server yap", "agent tanımla", "marketplace oluştur".

Akış

Tip netleştir → references/INDEX.md'den resmi URL → WebFetch ile code.claude.com/docs/...md çek → detay topla → frontmatter limiti uygula → schema'ya birebir uyan dosya yapısı → versiyon bump.

Öne çıkan

8 genişletme tipi, her biri için ayrı resmi URL. Schema uydurmama çekirdek kural. Mimari hatırlatmalar gömülü: klasörler .claude-plugin/ dışında, hook tipleri, MCP transport, plugin içi agent kısıtları.

project-init

Bir projeyi Claude Code ekosistemine entegre eder — CLAUDE.md/local, solo.yml, .gitignore eksiklerini tespit edip tamamlar, Vikunja/Solo/Obsidian entegrasyonlarını kurar.

Tetikleyici

"projeyi kur", "init", "setup", "CLAUDE.md oluştur". init.md oturum başı eksik tespit edince otomatik önerir.

Akış

Durum tespiti → manifest'ten stack çıkarımı → eksik listesi → toplu soru bloğu → stack'e uygun dosya üretimi → entegrasyon (vikunja project create, Solo, Obsidian klasör).

Öne çıkan

Obsidian klasörü yeni oluşunca obsidian-init otomatik tetiklenir; mevcut seçilince önce sorar. Override etmez, sadece eksik doldurur. Stack belirsizse durur ve sorar.

obsidian-init

Projeyi analiz edip vault'ta MOC + wikilink ilişkisel proje belleği oluşturur. Kendisi analiz yapmaz — obsidian-initializer orchestrator'ını çağırıp 5 alt agent'ı koordine ettirir.

Tetikleyici

"obsidian belleği oluştur", "obsidian init", /obsidian-init (--force ile yeniden üretir).

Akış

Obsidian Folder kontrol → mevcut index.md varsa --force iste → Task ile orchestrator → yazılan dosyaları göster.

obsidian-note

Tek seferlik bir bilgiyi (API key, sunucu, karar, komut, pattern) proje klasörüne kaydeder. obsidian-writer MODE: append çağırır — mevcut not varsa günceller, yoksa yeni Learnings notu.

Tetikleyici

"bunu obsidian'a not al", "obsidian'a ekle", "bunu kaydet", /obsidian-note.

Akış

Folder + index kontrol → CLI durumu → içerik topla → kategori tahmin et (api-key/server/decision/bug/command/pattern) → Task writer append → frontmatter özetiyle raporla.

Öne çıkan

Frontmatter'a category/confidence/source/last_verified + MOC'a wikilink. CLI kapalıyken fonksiyon kaybı yok (Grep fallback).

obsidian-audit

Vault sağlık denetimi — kırık wikilink, orphan, deadend notlar, eskimiş Learnings ve düşük güvenli bilgileri tespit edip raporlar.

Tetikleyici

"vault audit", "kırık linkleri bul", "obsidian denetle", /obsidian-audit [folder].

Akış

CLI kontrol → hedef klasör → paralel unresolved/orphans/deadends → last_verified audit (30 gün+ stale) → confidence audit → markdown rapor + aksiyon önerisi.

Öne çıkan

Sadece yapıyı değil bilgi tazeliğini de denetler — 6 aylık API key'in hala geçerli olup olmadığını sordurur.

obsidian-recall

Learnings notlarından birini rastgele veya kategori-filtreli okur ve özetler (spaced repetition — eski bilgileri tekrar gündeme getirir).

Tetikleyici

"rastgele not", "eski öğrenilenler", "geçmişten bir bilgi", /obsidian-recall [tag].

Akış

CLI + folder → filtre varsa obsidian tag yoksa random:read → notu yapılandırılmış özetle (konu/kategori/last_verified/güven/özet) → aksiyon öner.

Öne çıkan

obsidian-search'ten farkı: rastgele (search hedefli). /loop 4h /obsidian-recall ile otomatikleştirilebilir — sabah ritüeli, key rotasyon kontrolü.

obsidian-search

Vault'ta sorgu bazlı hedefli arama — aranan konuyu bulur, notları okur, sentezlenmiş cevap + wikilink kaynak listesi döner. Asıl işi obsidian-searcher agent'ına delege eder.

Tetikleyici

"obsidian'da ara", "X hakkında ne not almışım", "Y sorununu nasıl çözmüştük", /obsidian-search [sorgu].

Akış

Sorgu al → Folder'ı path filtresi için çıkar → CLI durumu → obsidian-searcher background çağır (QUERY/FOLDER/VAULT/CLI_ACTIVE) → sentez + kaynakları sun → güncelleme istenirse writer'a delege.

Öne çıkan

Agent salt-okur; yazma gerekince writer'a delege. Workflow kuralı gereği searcher her zaman background.

obsidian-doc-source

Dış kaynağı (URL/kütüphane/PDF/repo) global docs klasörüne sectioned API reference olarak dokümante eder. Proje folder'ı altında DEĞİL — tüm projeler aynı docs havuzunu paylaşır.

Tetikleyici

"bu kaynağı dokümante et", "API'yi dokümante et", "kütüphaneyi kaydet", /obsidian-doc-source <kaynak>.

Akış

Router ile tip tespit (GitHub→gh, web→WebFetch/crawl2md, library→ctx7, dosya→markitdown) → çakışma kontrol → kaynağın tamamını oku → temizle → bölümlere ayır (overview/auth/endpoints/...) → writer MODE: doc-source.

Öne çıkan

Çıktı proje-bağımsız global havuz — bir kez dokümante edilen Stripe API tüm projelerden erişilir. Library için Context7 zorunlu (bellekten yazmak yasak). Bounded extraction (>50 dosya/>500KB için sorar).

crawl2md

Bir web sitesini recursive crawl eder, içeriği markdown'a çevirir, web-scrape-cleaner ile temizler. İki aşamalı, her aşamada onay alır.

Tetikleyici

"siteyi crawl et", "URL'i markdown yap", /crawl2md <url> <dir> [--depth N] [--delay S].

Akış

uv kontrol → parametre parse → crawl onayı → uv run --script (PEP 723 inline deps, ephemeral venv) → rapor → temizleme onayı → cleaner agent.

Öne çıkan

Same-host kilidi — sadece aynı hosttaki linkleri izler. Delay zorunlu (rate-limit). Cleaner aggressive modda yedek önerir.

diji-log-search

Diji tabanlı b2c Django projelerde bir basket'in (UserBookingBasket) tüm yaşam döngüsünü prod AppLog tablosundan çıkarır, özetler, sorunu teşhis eder. Ref verilmezse hata mesajından/ekran görüntüsünden ilgili basket'i kendisi bulur.

Tetikleyici

"şu hata alındı son loglara bak", "bu hata mesajını araştır", "<ref> sepetini incele", "ödeme neden başarısız", "provider logları", /diji-log-search.

Akış

Ortamı CLAUDE.local.md'den oku → mod belirle (lifecycle/provider/payment/raw/find/diagnose) → scripts/diji_log_search.py'yi prod'a scp → manage.py shell < ile çalıştır (param env var ile) → LogReader sağlık kontrolü → kronolojik rapor.

Öne çıkan

find modu "mesaj → kod → basket → trace" zincirini tek komuta indirir. Marka/proje adı hardcode edilmez (domain her zaman CLAUDE.local.md'den). Çıktı /tmp'e yazılır (canlı credential içerebilir, Zed ile açılmaz).

worktree

Git worktree yönetimi — Claude Code native --worktree/EnterWorktree'ye delege eden wrapper. İzolasyon, paralel subagent, .worktreeinclude kurulumu, merged temizlik, PR akışı.

Tetikleyici

"worktree aç", "paralel çalışalım", "izole branch'te çalış", "worktree'den PR aç", /worktree.

Komutlar
  • new/enter/exit — worktree'ye gir/çık
  • list/status — tablo (ahead/behind)
  • pr — push + commit skill + gh pr create
  • clean --merged — merged'leri onaylı sil
  • setup — .worktreeinclude kurulumu
  • parallel N görev — N subagent izole kopyada
Öne çıkan

Native tool/flag üzerine ince koordinasyon. Dirty worktree asla otomatik silinmez. Commit mantığını DRY için commit skill'ine bırakır.

user-message

Müşteri/acenta tarafına kanal-agnostik mesaj üretir (ticket/email/panel/WhatsApp/SMS/push). Stil rehberini her çalıştırmada okur, onaya sunar, revize gelirse hem düzeltir hem stil dosyasına öğrenim işler.

Tetikleyici

"müşteri mesajı yaz", "acentaya mesaj", "zammad cevap", "duyuru yaz", "whatsapp mesajı", "sms yaz".

Akış

Stil rehberini oku (yoksa dur) → kanal belirle → bağlam topla → mesaj üret (self-check: teknik terim/iç link sızdı mı, kanal uzunluğu) → onay → revize ise yeniden üret + stil dosyasına ders ekle.

Öne çıkan

E-posta kanalı zengin HTML üretir (vurgulu tablo, kalın, liste) ve sistem varsayılan programıyla (open) açıp AskUserQuestion ile onay alır; diğer kanallar düz metin + zed --wait. Stil dosyası tek doğruluk kaynağı (bellekten yazmaz). Revize = öğrenim: her düzeltme kalıcı ders olur. Türkçe karakter ASCII karşılığı (s/sh) yasak.

full-translation
çeviri ajanları: sonnet

Django projesinde tüm dillerin eksik/fuzzy çevirilerini uçtan uca tamamlar — makemessages → po-cli analyze → dil başına paralel ajan çeviri → apply → compilemessages → commit. po-cli skill tek .po dosyası için; bu skill çok dil × çok domain (django/djangojs/djangof7) matrisini orkestre eder.

Tetikleyici

"tam çeviri akışı", "tüm dilleri çevir", "i18n sync", "translation workflow", /full-translation.

Akış (7 adım)

Repo + mobil pull (koşulsuz zorunlu — mobil HEAD=origin/main doğrulanmadan Adım 1'e geçilmez; atlanırsa F7 metinleri sahte "0 untranslated" verir) → makemessages (3 domain) → po-cli --json analyze → Workflow: dil başına 1 ajan paralel çeviri → dry-run validate + tek onay + apply + perl temizlik → compilemessages + re-analyze (yakınsama) → commit skill.

Öne çıkan

Fuzzy msgstr'ye güvenmez (msgid'den sıfırdan), placeholder/slug/marka kuralları ajan prompt'una gömülü, plural (nplurals) tablosu dile göre. po-cli'yi tekrar etmez, ona dayanır.

Agents · 8 adet

İzole context'te çalışan uzman alt-agent'lar

Mimari: skill'ler ince tetikleyici, ağır iş agent'lara delege. Obsidian ekosistemi merkezde.

obsidian-writer
4 mod · merkezi yazma noktası · model: sonnet

Vault'a MOC + wikilink yapısıyla not yazan merkezi agent. Tüm Obsidian yazma işlemlerinin tek noktasıdır; frontmatter, ilişkisel wikilink ve dedup kurallarını her modda korur. Tools: Read/Write/Edit/Bash/Glob/Grep.

4 Mod
  • init — orchestrator'dan; 4 alt agent çıktısından index.md MOC + Stack/Architecture/Recent-Activity/README-Summary üretir.
  • append — Stop hook/ad-hoc; hook kaynaklı otomatik kayıtlara kayda değerlik guard'ı uygular (dar kritere girmeyen veya vault'ta zaten olan içerik yazılmadan SKIPPED/DUPLICATE döner; açık kullanıcı isteği ve skill onayı guard'ı atlar — kriterlerin kanonik kaynağı bu agent'tır), sonra içeriği konulara ayrıştırır (bağımsız konular → ayrı notlar, ilişkili konular → tek notta tutup etiketle/cross-link), kalıcı bilgi → Learnings/{topic}.md, zamanlı → Journal/{date}.md; MOC'a wikilink + last_verified.
  • doc-source — dış kaynağı global docs/<source>/ altına multi-file API reference; global docs MOC'unun tek sahibi.
  • update — mevcut dosyada hedefli düzenleme (madde işaretle, section güncelle); "override etme"nin kasıtlı istisnası, vault-içi + .md güvenlik sınırı, en-az-değişiklik ilkesi.
obsidian-initializer
orchestrator · model: sonnet

Projeyi tarayıp vault'ta MOC + wikilink proje belleği oluşturan orchestrator. Kendisi içerik üretmez — 5 alt agent'ı koordine eder. Tools: Task/Read/Glob/Grep/Bash.

Nasıl çalışır

Hedef klasör hazırla → 4 alt agent'ı (stack-detector, arch-mapper, readme-extractor, recent-activity) tek mesajda paralel Task ile çağır → raporları writer MODE: init'e ver → sonuç raporu.

obsidian-searcher
salt-okur arama · model: haiku

Vault'ta sorgu bazlı arama yapan salt-okur agent. Aranan konuyu bulur, notları okur, sentezler, wikilink kaynaklarıyla bağlar. Hiçbir şey yazmaz. Tools: Bash/Read/Grep/Glob.

MOC → search → context → sentez
  • 0. CLI ön-kontrol — kapalıysa filesystem fallback.
  • 1. Önce MOC — index.md'de eşleşen wikilink varsa notu direkt oku.
  • 2. BM25 search — tek/az terim (çok kelime AND'lenip boş döner), TR↔EN sinonim, path filtresi.
  • 3. Context — search:context ile eşleşen satır + çevresi.
  • 4. Oku + sentezle — 1-3 notu tam oku, backlinks keşfet, birleştir.
Çıktı

Sentezlenmiş paragraf (kod yolu/sayı/prosedür) + ### Kaynaklar wikilink listesi. Her iddia bir wikilink'e dayanmalı (uydurma yok); bulunamazsa dürüstçe söyler.

stack-detector
model: haiku

Tech stack tespiti (dil, framework, paket yöneticisi, major bağımlılıklar). Manifest dosyalarını okur, framework'ü işaret dosyalarından çıkarır, paket yöneticisini lock file'dan bulur. Belirsizse "Tespit edilemedi" — varsayım yapmaz. Tools: Read/Glob/Grep/Bash. obsidian-initializer çağırır.

arch-mapper
model: haiku

Mimari haritalama (top-level dizinler, backend/frontend/admin ayrımı, katmanlar). Klasörleri rolüne göre sınıflar, Django app'lerini/monorepo paketlerini listeler, dosya sayılarıyla yapı çıkarır. node_modules/.venv/dist atlanır. Tools: Read/Glob/Grep/Bash. obsidian-initializer çağırır.

readme-extractor
model: haiku

README özeti — proje amacı, setup adımları, ana özellikler ve komutlar. README yoksa boş rapor. README'de olmayan bilgi eklenmez (varsayım yok). Tools: Read/Glob. obsidian-initializer çağırır.

recent-activity
model: haiku

Son 30 commit analizi — hot area'lar, en çok değişen dosyalar, aktif contributor'lar. git log -30 ile dosya/klasör/contributor frekansı sayılır. Git repo değilse veya az commit varsa uyarlanır. Tools: Bash/Read. obsidian-initializer çağırır.

web-scrape-cleaner
model: haiku

Ham scrape markdown'ındaki gürültüyü temizler (nav/footer/cookie banner/reklam, boş başlık, link gürültüsü). Maliyet-optimize haiku modelinde çalışır. Tools: Read/Edit/Write/Glob/Grep/Bash.

Nasıl çalışır

Keşif (örnek dosyada gürültü tespit) → tüm dosyaları concat + uniq -c ile 5+ dosyada tekrar eden satırları sil → aggressive/conservative temizleme → fark >%5 ise üstüne yaz (orijinal silinmez) → başlık hiyerarşisi + link sadeleştirme.

Öne çıkan

Asla scrape etmez (sadece mevcut .md düzenler), asla git commit yapmaz, KEEP listesine dokunmaz. İstenirse .original/ yedeği bırakır.

Tek komutla kurulum

Tüm stack'i tek komutla kuran idempotent installer. setup.sh önce uv yoksa kurar, sonra setup.py'yi rich/questionary bağımlılıklarıyla ephemeral venv'de çalıştırır (global kirletmeden).

Ne kurar

settings.json + statusline, Homebrew prerequisites, Python CLI araçları (uv tool), Obsidian vault + CLI, local MCP repo'ları, vikunja-cli, zammad-cli, po-cli, Playwright CLI, Context7 CLI, Claude Code MCP server'ları ve interaktif credential prompt'ları.

Çalıştırma
./setup.sh              # uv kurup başlatır
# idempotent — tekrar çalıştırılabilir, mevcut olanı atlar
Obsidian
vault + CLI · proje belleği

Nedir: Markdown tabanlı, yerel bir bilgi tabanı uygulaması; notları [[wikilink]] ile birbirine bağlayıp graf oluşturur. Resmi CLI (obsidian komutu) ile programatik okuma/yazma/arama sağlar.

Ne işe yarar: Proje-spesifik kalıcı bilgiyi (mimari karar, bilinen bug, credential, çalışan komut) kaybetmeden saklar ve BM25 ile aranabilir kılar.

Plugin'deki rolü

Plugin'in kalbi. 5 skill (init/note/search/recall/audit) ve 3 agent (writer/initializer/searcher) doğrudan Obsidian üzerine kurulu; 4 hook (context/daily/userprompt-hint/stop-daily) vault ile konuşur. Vault ~/Documents/ObsidianVault/<proje>/ altında MOC + wikilink yapısındadır. CLI kapalıyken filesystem fallback'e (Read/Grep) düşer.

Vikunja
vikunja-cli · görev yönetimi

Nedir: Açık kaynak, self-hosted bir görev/proje yönetim aracı (Todoist/Asana alternatifi). Plugin vikunja-cli binary'si üzerinden REST API'sine bağlanır.

Ne işe yarar: Görev listeleme, oluşturma, durum güncelleme, atama ve yorum — terminalden çıkmadan iş takibi.

Plugin'deki rolü

init.md kuralı: kullanıcı bir sorun bildirince ilgili Vikunja görevi aranır, yoksa oluşturma teklif edilir. commit skill'i: değişiklikle uyuşan görev tespit edilip kapatılabilir. vikunja-cli skill'i doğrudan task yönetimi sağlar.

Yapılandırma
~/.zshrc → VIKUNJA_API_URL + VIKUNJA_API_TOKEN
CLAUDE.local.md → Vikunja proje ID
Solo
CLI · process yönetimi

Nedir: Geliştirme process'lerini (backend, frontend, worker vb.) yöneten bir araç; solo.yml ile tanımlanan komutları başlatır, durdurur, yeniden başlatır ve çıktılarını takip eder. solo CLI binary'si Solo app'in HTTP control plane'ine bağlanır (solo doctor ile doğrulanır).

Ne işe yarar: Tekrar eden shell komutlarını (build + run) tek yerde toplayıp Claude'un bunları başlatıp loglarını okumasını sağlar.

Plugin'deki rolü

soloterm.md kuralı: tekrar eden komutlar için mutlaka solo.yml oluşturulur ve komutlar Solo üzerinden çalıştırılır. Process başlatma/durdurma/restart solo processes / solo commands CLI komutlarıyla yapılır — ancak process logu/çıktısı CLI'de görünmez (hata teşhisi için GUI veya log dosyası workaround'ı gerekir). (Bu plugin repo'sunun kendisi için aktif değil — çalışan process yok.)

WhatsApp
whatsapp-mcp · mesaj kanalı

Nedir: WhatsApp Web bridge'i üzerinden mesaj gönderme/okuma sağlayan bir MCP server (whatsapp-mcp, stdio). QR kod okutulup bridge başlatılarak bağlanır.

Ne işe yarar: Sohbetleri listeleme, mesaj arama, kişi bulma ve mesaj/dosya/ses gönderme — Claude'dan doğrudan WhatsApp iletişimi.

Plugin'deki rolü

user-message skill'inin çıktı kanallarından biri. Skill müşteri/acenta mesajını kanal-agnostik üretir; WhatsApp seçilirse mcp__whatsapp__* araçlarıyla gönderilir (skill mesajı üretir, gönderimi entegrasyon yapar).

Google Workspace
Gmail · Calendar · Drive (claude.ai MCP)

Nedir: claude.ai üzerinden sağlanan resmi Google MCP server'ları — Gmail (e-posta), Google Calendar (takvim) ve Google Drive (dosya). HTTP MCP olarak bağlanır, OAuth ile yetkilendirilir.

Ne işe yarar: E-postaları okuma/arama/gönderme, takvim etkinliklerini görüntüleme/oluşturma ve Drive dosyalarına erişim — Claude'dan doğrudan Google hesabıyla çalışma.

Plugin'deki rolü

user-message skill'inin e-posta kanalı Gmail ile gönderilebilir; takvim/dosya entegrasyonu genel asistan akışlarını besler. Üç servis de claude.ai üzerinden gelir — ayrı binary kurulumu gerekmez.
